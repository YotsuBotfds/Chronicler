### Azgaar/Fantasy-Map-Generator -> M58 Agent-Level Trade

**Problem in milestone:** M58 needs a network topology for merchant route selection. The current abstract trade model uses boundary-pair connections, but agent-level merchants need richer pathfinding with terrain-aware costs and natural topology.

**Technique:** Urquhart graph (pruned Delaunay triangulation) for trade route topology. The Urquhart graph removes the longest edge from each Delaunay triangle, producing a sparse planar graph that resembles natural transportation networks -- roads follow valleys, avoid mountains, connect population centers. Combined with Dijkstra flood fill using domain-specific cost functions (different costs for different terrain types, culture groups, naval routes).

**Integration:**
- **Type:** spec enrichment for M58
- **Target file(s):** `src/chronicler/economy.py` (route topology), future `trade_routes.py`
- **Estimated scope:** ~80 lines Python for Urquhart pruning from region adjacency graph + terrain-weighted Dijkstra. Route topology computed once at world gen, updated on conquest/infrastructure events.
- **Dependencies:** M55 spatial positioning (for meaningful terrain costs)

- How the generator handles chokepoints (narrow passages between mountains become critical trade nodes)

**Technique:** 3D spatial SEIR ABM with R0 calculations. The key extractable pattern: `beta_local = beta_base * (neighbors_within_radius / expected_density)`. Transmission probability scales with the ratio of actual to expected local density. Dense clusters amplify transmission; sparse areas suppress it.

- **Type:** spec enrichment for M55's disease downstream effect
- **Target file(s):** `src/chronicler/emergence.py` (pandemic mechanics), future Rust disease tick
- **Estimated scope:** ~30 lines. Replace per-region uniform `endemic_severity` application with density-modulated severity: `effective_severity = endemic_severity * (local_density / mean_density).min(3.0)`. Local density from M55 spatial hash cell population.

## Cross-Cutting Patterns

### Pattern 1: Asymmetric Decay Rates
**Applies to:** M50 (relationships), M71 (aggressive expansion), M66 (legitimacy)

Negative experiences (betrayal, conquest, persecution) should decay 3-5x slower than positive experiences (peace, trade, cooperation). This pattern appears in: Unciv's asymmetric modifier decay (4:1 ratio), M48's half-life parameterization (persecution at 50 turns vs prosperity at 20 turns), and EU4's aggressive expansion tracker. The 3-5x ratio is empirically robust across behavioral economics research (loss aversion).

**Implementation:** Every system that tracks bilateral sentiment or reputation should use `NEGATIVE_DECAY_RATE = POSITIVE_DECAY_RATE * ASYMMETRY_FACTOR` where `ASYMMETRY_FACTOR` is in [3.0, 5.0]. `[CALIBRATE]` per system.

### Pattern 2: Proportional Restoration with Linear Decay
**Applies to:** M49 (needs), M63 (institutional legitimacy), M66 (political capital)

The combination of linear decay (constant drain) with proportional-to-deficit restoration (diminishing returns near cap) produces stable equilibria without oscillation. M49 spec already uses this pattern. Extend to institutional legitimacy (M63: legitimacy decays linearly from enforcement costs, restores proportionally from ruler actions) and political capital (M66: capital drains from governance costs, restores proportionally from victories/ambitions).

**Implementation:** `value += RESTORE_RATE * condition * (1.0 - value); value -= DECAY_RATE;`

### Pattern 3: Two-Tier Decision Models
**Applies to:** M53 (Fermi function), M58 (merchant route selection), M60 (war decisions)

Hard threshold gate first (am I stressed enough to even consider action?), then probabilistic/utility-based comparison of alternatives. Prevents computational waste (agents not under stress skip evaluation entirely) and prevents noise-driven decisions (agents near equilibrium need a significant utility difference to switch). From abm-ccid, already cited for M53. Extend to M58 (merchants only evaluate routes when profit exceeds `MIN_PROFIT_THRESHOLD`) and M60 (war only considered when EU > 0).

### Pattern 4: Validation Oracles at Every Tuning Pass
**Applies to:** M53, M61, M67, M72

Each tuning pass should have: (1) State hash determinism check, (2) Macro-pattern validation targets from external sources (psychohistoryML era distributions, GovSim commons failure rates, Artificial Anasazi settlement patterns), (3) Per-system health metrics (distribution not degenerate, activation fraction within targets), (4) Narrative diversity via 6-arc classification.

### Pattern 5: Edge-Resident State for Transit
**Applies to:** M58 (goods in transit), M59 (information in transit), M60 (armies in transit)

When entities travel between nodes, they exist on the edge during transit. `TransitEntity { type, origin, destination, turns_remaining }`. Prevents double-counting (entity not at origin or destination during transit). From amzn/supply-chain (goods), applicable to M59 (information packets), M60 (marching armies). Unified data structure, different entity types.
